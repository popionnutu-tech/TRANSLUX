# Moneyball — редизайн на credibility-shrinkage (Tier 1)

## Контекст / зачем
`moneyball_recompute()` (ночной pg_cron `moneyball-nightly`, 02:00 UTC) считает качество водителей
vs нормы по 6-мерным ячейкам (маршрут×направление×остановка×квартал×тип-дня×вместимость).
Две беды:
1. **Сломан:** падает каждую ночь ≥10 дней на рейсе без водителя (`driver_id=NULL` в `analytics_trip_scores`,
   колонка NOT NULL). Функция начинается с `TRUNCATE` → откат → данные заморожены.
2. **Слабая методика:** «сырые средние» по очень мелким ячейкам (часто 1–3 рейса) → шум; нет усадки,
   нет веса по выборке, нет значимости; нулевые/выбросовые рейсы искажают.

Deep-research (actuar credibility, D. Robinson EB, openWAR, statsmodels; 25/25 проверенных утверждений)
→ правильный приём — **credibility = empirical-Bayes shrinkage**: усаживать оценку к более грубой/флотской
средней с весом `Z = n/(n+K)`. Один приём чинит шум, усадку и вес по выборке. Это closed-form → **чистый SQL**.

## Подход (формулы)
Гауссова credibility (Bühlmann-Straub) — distribution-light, SQL-native. (Точные Poisson/NB-модели под
счётные данные — отдельный Tier-2 рефайнмент; механизм усадки тот же.)

**A. Иерархические нормы со усадкой (чинит шум/покрытие, проблемы №1,2).**
Для каждой 6-D ячейки: `baseline = Z_cell·μ_cell + (1−Z_cell)·μ_parent`, где
- `μ_cell` — среднее пассажиров ячейки, `n_cell` — число рейсов;
- `μ_parent` — грубее: маршрут×направление×остановка×квартал (без day_type/capacity); если и там мало —
  выше: маршрут×направление×остановка;
- `Z_cell = n_cell/(n_cell + K_b)`, `K_b = EPV/VHM` (EPV = средняя внутриячейковая дисперсия пассажиров,
  VHM = дисперсия средних ячеек − EPV/n̄), оценивается глобально по Bühlmann в SQL.
→ нормы стабильны даже при 1–3 рейсах, дыр в покрытии нет.

**B. Усадка балла водителя (чинит «3≈30 рейсов», проблема №3).**
Балл водитель×маршрут: `score = Z_drv · d_raw`, где `d_raw` — средневзвешенное отклонение (как сейчас),
`Z_drv = n/(n + K_d)`, `K_d = (внутриводительская дисперсия отклонений)/(межводительская дисперсия)`.
→ мало рейсов → тянем к 0 («как ожидалось»); много → почти своя цифра.

**C. Доверие/значимость (проблема №3).** Closed-form ст.ошибка credibility: `SE ≈ Z_drv·sqrt(внутр.дисп/n)`.
95% CI = `score ± 1.96·SE`; `is_significant = CI не включает 0`. (Bootstrap-CI — Tier 2.)
В ранге: водителей с CI, перекрывающим 0, помечаем «статистически неотличим от среднего».

**D. VORP в леях (проблема №5).** На SHRUNK-балле: `vorp_lei = (score/100)·baseline_pax·n_trips·lei_per_pax`.
База сравнения — **«средний водитель»** (норма = средний), это и есть текущий смысл и понятнее для бизнеса
(«на сколько лей больше/меньше среднего водителя на тех же рейсах»). [РЕШЕНИЕ владельца: средний vs нижний квартиль.]

**E. Гигиена данных (проблема №4 + баг).** Исключить из скоринга: рейсы с `driver_id IS NULL`;
завершённые сессии с 0 пасс. И 0 лей (артефакты). [РЕШЕНИЕ владельца подтвердить фильтр.]

## Изменения
- **Функция `moneyball_recompute()`** переписывается: шаги A–E. Те же таблицы, новые формулы.
- **Схема `analytics_driver_route_scores`** + колонки: `shrunk_deviation_pct`, `credibility_z`, `ci_low`,
  `ci_high`, `is_significant`; `vorp_lei` считается по shrunk. `avg_deviation_pct` (сырое) остаётся как было
  (обратная совместимость дашборда). Параметры K_b/K_d → в `analytics_cron_runs` (аудит).
- **Дашборд** (apps/admin .../analytics/moneyball): сначала проверить путь чтения (views `v_moneyball_*`
  vs прямые таблицы — verify в импле, починить если отсутствуют), затем показать shrunk-балл + пометку
  «значимо/неотличим». Цифры берём из новых колонок.
- **Миграция** (packages/db/migrations/NNN): схема + `CREATE OR REPLACE FUNCTION`. Применить, прогнать
  `SELECT moneyball_recompute();` вручную, проверить успех + вменяемость чисел + `analytics_cron_runs.status='success'`.

## Проверка (как убедимся)
1. Ручной прогон `moneyball_recompute()` завершается без ошибки null-driver, `analytics_cron_runs` = success.
2. Нормы: у мелких ячеек baseline тянется к родителю (спот-проверка Z_cell→0 при n=1).
3. Баллы: водитель с 3 рейсами усажен к 0 сильнее, чем с 30 (спот-проверка).
4. Значимость: часть водителей помечены «неотличим» (CI⊃0).
5. Дашборд читает новые цифры; страницы не пустые.

## Ограничения (честно)
- Гауссова аппроксимация на счётных данных приемлема при этих величинах; точные Poisson/NB — Tier 2.
- «Replacement=средний» — бизнес-выбор; нижний квартиль легко включить.
- Перекрёстные эффекты маршрут×водитель и bootstrap-CI — Tier 2 (офлайн statsmodels), не в ночном SQL.

## Реализация (факт) и ревизии по ревью
Реализовано 4 миграциями (применены к проду `zqkzqpfdymddsywxjxow`, каждая прошла architecture-guardian + performance-reviewer):
- **107** — редизайн `moneyball_recompute()`: гигиена (completed, capacity≠1, отсев «0 лей И 0 пасс.», `driver_id IS NOT NULL` — фикс краша), иерархические нормы со усадкой (Bühlmann K_b), усадка балла водителя (K_d), CI/значимость, VORP на shrunk. Колонки в `analytics_driver_route_scores` (shrunk/z/ci/is_significant) + K_b/K_d в `analytics_cron_runs`. Прогон: success, ~15с, k_b=2 (пол), k_d=50 (потолок — слабый сигнал между водителями → сильная усадка, это норма).
- **108** — РЕВИЗИЯ по ревью: ⚠️ **`avg_deviation_pct` теперь ХРАНИТ усаженный балл** (его читают ВСЕ потребители — рейтинг/VORP/рекомендации/insight), сырой ушёл в `raw_deviation_pct` (исправлена несогласованность «один водитель — разные цифры на разных экранах»). `is_significant` требует **n≥5** (честность на малых выборках). `REVOKE EXECUTE … FROM service_role` (защита 15-сек функции от 8-сек RPC-таймаута).
- **109** — `v_moneyball_recommendations`: «срочно» на статзначимости + занос 3 вью Moneyball (recommendations/group_recommendations/segments) в репозиторий (раньше жили только в живой БД).
- **110** — `urgent_change` = ГИБРИД: значимость-только давала 0 срочных (честная усадка → почти ничего не 95%-значимо), поэтому urgent = есть лучший (разрыв>2п.) И опытный (≥10 рейсов) И (значим ИЛИ <−1.5%). Даёт 4 реальных кейса (Goreaci ~3608 lei/мес и др.).

## Открытые follow-up (требуют решения владельца)
1. **Per-route рекомендации не выведены в UI.** Страница «Recomandări» читает ГРУППОВУЮ вью (`v_moneyball_group_recommendations`), а `v_moneyball_recommendations` (где «срочно»/гибрид) **не читает ни один экран**. Логика верна и версионирована, но эффект не виден — нужен кусок фронта, если хотим показать per-route замены (Goreaci и т.д.).
2. **Дубли водителей в `drivers`.** 14 ФИО имеют 2–3 `id`. Полирует рейтинг/VORP везде; делает 1 из 4 «срочных» (Crestianov) самопротиворечивым. НЕ сливать вслепую — одинаковое ФИО может быть тёзками; нужна проверка по владельцу (телефон/факт). 
3. **Tier-2 (по желанию):** точные Poisson/NB + bootstrap-CI офлайн; усадка сегментного балла (`v_moneyball_segments` пока сырой); badge значимости во фронте (после п.1).
